Cypress

I have built end to end automation frameworks in Cypress three times:

At VMWare Carbon Black, I ran the POC of Cypress alongside another engineer. Once we proved out the framework suiting our needs, I spearheaded automating our regression test cases which had grown to cover 3 days worth of manual testing. Upon completion, we wrote documentation around best practices and how to write automation for our developers to contribute as well.

At Brio Systems, I presented Cypress to the CTO in my first week there as they had no automation coverage for legacy software and we were about to begin development of a new software. I did this by taking a piece of our legacy software and automating it in Cypress, then talking through the possibilities of extending the suite. I got buy-in and began development of our automation framework. While the initial ground work was happening with our new software, I was able to automate the process of initial contact with our software to getting notified of lab results being ready by SMS, and then using a time-based-one-time-passcode to access the results. This was my first exposure to more complex integrations with Cypress, using Twilio to manifest the SMS portion and access the TBOTP via API. As development continued with the new software, I was also able to implement accessibility testing through AXE and form statefulness by exposing the redux store.

At Hometap, Cypress was already the chosen language when I joined but it was implemented poorly. It was done through page-object modeling which is not a Cypress best practice, and it was heavy on aliasing GraphQL calls and mutations in a very challenging to access manner. There was reliance on QA specific api endpoints and bypassing a lot of the production workflow which required a large refactor. I brought up the idea of persona driven testing and having end-to-end tests cover use cases to drive the software from end to end through the eyes of various stakeholders. We were also unable to execute these tests on our staging environment as we needed access to a outside email inbox. I set up access to a google user group via Google OAuth in our Cypress repository and created a recursive polling function to monitor the inbox and avoid arbitrary waits. We were able to piece the two ideas and refactors together to get comprehensive end to end tests running on our staging environment during Release Candidate testing, ensuring much higher quality all around. I also helped to introduce component level testing via Cypress mocking out our backend and using only front end for more unit/integration level testing that was missing in our MR pipelines.

Postman

I have always built out individual components of our automation suites in a Postman collection. This helps to debug and attian information while developing future automation or harden manual tests. An example of this is at Hometap:

We have automation that spans Homeowner dashboard view and tasks all the way through internal Application Specialists and Underwriters seeing an investment through. I created a Postman collection that authenticates as any user, uploads any accepted document type, completes any smart task, gets track info or document id's, generates legacy data for migration testing, and much more. This helps the QA team ensure stability of leveraging API's consistently and being able to pull that into our automation once figured out.

Python

At VMWare Carbon Black, the bulk of our API testing was written in Python. When I joined I asked to contribute to that repository and help grow it. The team I was embedded on was a backend team so it fit well. I was able to call Atomic Red Team scripts via Python and then add tests to validate alert counts incrementing, then asserting the appropriate metadata that should be associated with those alerts. We had different threat reports, so there was an endless amount of ART tests to call and validate, this work was a constant throughout my time there.

At Hometap, all of our unit testing automation was written in Unit Test. I wrote and proposed a TRD to our release and development guilds that we audit and update our framework to PyTest. The change was approved, and I am leading the effort to run a gap analysis of our code coverage for unit testing, and pivot our framework to PyTest as we add more tests and update existing ones.
